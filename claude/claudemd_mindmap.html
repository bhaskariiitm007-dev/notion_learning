What Is claude.md?
A markdown file that becomes Claude's permanent memory
LLMs have ZERO native cross-session memory — claude.md is the only bridge
★
Persistent system prompt — automatically injected at the start of every Claude Code session
1
Stores project rules — coding conventions, architectural decisions, naming standards
2
Two creation methods — write it manually or generate a base version using /init slash command
3
Lives in project root — stored alongside your code so every team member and session shares identical context
4
Living document — update it as the project evolves; treat it like code, not a one-time setup file
💡
⭐ Why It Exists
Without claude.md, Claude Code starts from zero every session. You'd re-explain your entire project context repeatedly — a compounding tax on every conversation.

The .claude Folder
Claude Code's toolbox — two distinct scopes
📁
Project-Level
Shared via repository with the whole team. Project-specific config, shared commands, team conventions.
  • Committed to Git
  • Team-wide rules
  • Scoped to this repo
🏠
Global User Folder
Stored in home directory — private and personal, not committed to Git.
  • Private to you
  • Cross-project settings
  • Personal skills & cmds

Common Mistakes to Avoid
What makes claude.md useless or counterproductive
!
Bloated file — exceeding ~200 lines buries important rules in noise; AI context has limits
!
Treating it as static — never updating means Claude Code works from stale, incorrect context
!
One giant root file — dump all subdirectory rules into one file instead of using folder-specific files
!
Ignoring memory.md — auto-memory can accumulate incorrect patterns; periodically review and prune it
!
Missing global folder — forgetting to set up ~/.claude loses personal cross-project settings
📄 Bloated File⏸️ Stale Context🗂️ Monolithic Rules🧠 Unchecked Memory
