What Is the Context Window?
Your model's finite, paid, shared working memory
LLMs are STATELESS — no native cross-session memory
★
Token-limited RAM — everything the model sees right now must fit within a hard token ceiling
1
Full history every call — LLMs are stateless; every request resends the entire conversation from scratch
2
History compounds fast — tool outputs, code blocks, and back-and-forth drain the budget rapidly
3
Quality degrades at the limit — hitting the ceiling causes early context to drop, producing inconsistent responses
4
Cost scales linearly — every resent token is billed; bloated context directly increases API costs
⚠️
Root Problem: Statelessness
A 50-message session resends all 50 messages on every call. The "memory" isn't free — it's paid for in tokens every single request.

Four Tools for Context Optimization
Your complete arsenal for keeping the context window healthy and efficient
✂️
/compact — Manual
Summarize history into a compact digest on demand. You decide when to compress — preferred over auto-compaction for full control.
🤖
Sub-Agents
Parallel sub-agents each run in their own isolated context window. No cross-contamination, no token sharing.
🚫
.claudeignore
Block large irrelevant files from ever entering context. Modeled after .gitignore — stops token waste at the source.
🎯
Focused Prompts
Clear, single-purpose prompts consume far fewer tokens than rambling multi-part requests.
✂️ /compact🤖 Sub-Agents🚫 .claudeignore🎯 Focused Prompts
✂️
/compact — Manual vs Auto
Why manual compaction always beats automatic for serious work
★
Manual is preferred — run /compact proactively before hitting the limit, not reactively after degradation
1
Auto fires too late — fires at an arbitrary threshold, potentially summarizing context you still actively needed
2
Summary replaces raw history — distills conversation into a concise block, freeing all tokens held by raw turns
3
You control the timing — compact after each major feature, each debug session, or any large code block
✂️ /compact👁️ Manual Control⚠️ Avoid Auto-Only
🤖
Sub-Agents — Horizontal Scaling
Isolated context windows that scale complexity outward
★
Own isolated context — each sub-agent starts fresh; parallel tasks don't compete for a shared token pool
1
Perfect for independent tasks — writing tests for module A and docs for module B simultaneously, zero contamination
2
No bleed into main session — sub-agent outputs don't bloat your primary context unless you explicitly import them
3
Decomposes large features — tasks too big for one context split across sub-agents on isolated sub-problems
💡
Mental Model
Sub-agents are like opening new browser tabs — each has its own fresh session. The parent session stays clean; only pull results back when you need them.

Context Drain Mistakes
What silently eats your token budget alive
!
Marathon sessions — one session for an entire project; context bloats with irrelevant history and quality crumbles
!
No .claudeignore — build artifacts, logs, and node_modules silently drain thousands of tokens every session
!
Auto-only compaction — fires too late, may discard context you were actively using at the moment
!
Vague multi-part prompts — rambling requests force broad reasoning across irrelevant scope, inflating token usage
!
GUI over terminal — GUI plugins omit power features: memory editing, hooks, and full context-management control
🏃 Marathon Sessions📁 No Ignore File⚠️ Auto-Only💬 Vague Prompts
